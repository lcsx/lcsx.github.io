# rest

# 客户-服务器

  1. 关注点分类
  2. 服务端专注数据存储，提升了简单行
  3. 前端专注用户界面，提升了可移植性（可移植性，过去前端还包含存储服务器的信息，这样就限制了代码只能运行在某个特定环境，新增电脑浏览器，手机里都可以）

# 无状态

  1. 所有的客户会话信息都保存在客户端
  2. 每次请求必须包括所有信息，不能依赖上下文
  3. 服务端不用保存会话信息，提升了简单性（代码简单了，服务端不用保存会话等）,可靠性（软件稳定程度，从故障中能快速恢复的正常能力），可见性（接口可见）

# 缓存cache

  1. 所有服务端响应都要被标记为可缓存或不可缓存
  2. 减少前后端交互，提升性能

# 统一接口（最重要）

  1. 接口设计尽可能统一通用，提升简单性，可见性
  2. 接口与实现解耦，使前后端可以独立开发迭代

# 分层系统

  1. 每层只知道相邻一层，后面隐藏的就不知道了
  2. 客户端不知道是和代理还是真实服务器通信
  3. 其他层 ：安全层，负载均衡，缓存层等

# 按需代码

  1. 客户端可以下载运行服务端传来的代码
  2. 通过减少一些功能，简化了客户端

# 请求设计规范

  1. URI使用名词，尽量用复数，如/users
  2. URI使用嵌套表示关联关系,如/users/12/repos/5
  3. 使用正确的HTTP方法，如GET/POST/PUT/DELETE
  4. 不符合CRUD的情况：POST/action/子资源

# 响应设计规范

  1. 查询
  2. 分页
  3. 字段过滤
  4. 状态码
  5. 错误处理
